## What is this project

**Gest2A3Eco** is a Python/Tkinter desktop application (Windows) for:
1. Generating `suenlace.dat` files (binary A3ECO accounting format) from Excel bank statements, issued invoices, and received invoices.
2. Creating and managing issued invoices internally (PDF generation, series numbering, customer master).

Entry point: `python main.py`

## Development commands

```bash
# Setup
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt

# Run
python main.py

# Build .exe
pyinstaller Gest2A3Eco.spec
# Output: dist/Gest2A3Eco/Gest2A3Eco.exe
```

No automated tests exist. Use the smoke test checklist in `CONTEXT.md` for manual validation.

## Architecture

Layered MVC:

```
views/ui_*.py          → Tkinter UI screens
controllers/*.py       → Business logic coordinators (navigation hub: app_controller.py)
procesos/*.py          → Domain logic: generate A3ECO binary records
models/gestor_sqlite.py → SQLite data access (source of truth)
models/facturas_common.py → Shared invoice model, date normalization, A3 record renderers
utils/                 → Config I/O, account validation, number formatting
services/              → Auth, secured gestor wrapper, Excel mapping, A3 binary importers
```

**Startup flow:** `main.py` → login → `UISeleccionEmpresa` → company selection → `AppController` → Notebook dashboard (tabs: Plantillas, Facturas emitidas, Generar ficheros).

**Data storage:** SQLite at `plantillas/gest2a3eco.db` (created on first run). Main tables: `empresas`, `bancos`, `facturas_emitidas`, `facturas_recibidas`, `facturas_emitidas_docs`, `terceros`, `terceros_empresas`, `usuarios`, `usuarios_empresas`, `plan_cuentas`. The legacy `GestorPlantillas` JSON manager is kept only for initial seeding.

**Authentication:** `services/auth_service.py` (scrypt hashing). `services/secured_gestor.py` wraps `GestorSQLite` with role/permission checks. Roles: ADMIN, EMPLEADO, CLIENTE. Per-company permissions: NINGUNO/LECTURA/ESCRITURA.

## A3ECO binary format details

- `models/facturas_common.py` is the central renderer for all A3ECO record types.
- Bank movements: type 0, 512 bytes/record.
- Issued invoices: type 1 (header) + type 9 (detail), 256 bytes — generated by `procesos/facturas_emitidas.py`.
- Received invoices: type 1/2 (header) + type 9 (detail), 254 bytes — generated by `procesos/facturas_recibidas.py`.
- Date field `_fecha_yyyymmdd` accepts date/datetime, pandas.Timestamp, Excel serials, and string formats; invalid dates → `"00000000"`.
- Account length is configurable per company (`empresas.digitos_plan`, default 8). Always respect this when building subcuentas.
- Concept matching in bank templates uses `fnmatch` wildcard patterns (`*`); first match wins, fallback to `subcuenta_por_defecto`.

## Key business rules

- **Subcuenta construction (issued):** `cuenta_cliente_prefijo` + NIF digits, adjusted to `digitos_plan`.
- **Subcuenta construction (received):** priority: `_cuenta_tercero_override` > `_usar_cuenta_generica` flag → `cuenta_proveedor_por_defecto` > Excel column > prefix + NIF.
- **Invoice grouping:** rows are grouped by `Numero Factura Largo SII` if present; otherwise by `Serie` + `Numero Factura`. Each group → one header + N detail lines.
- **Invoice numbering:** `serie_emitidas` + zero-padded counter (`siguiente_num_emitidas`) stored in `empresas` table.
- **Fiscal year:** stored per company and replicated on create/copy of templates, third parties, and invoices.

## Conventions

- Language: Spanish (code identifiers, UI, comments). Keep new files ASCII-safe (no accented chars in source).
- Layer separation: UI in `views/ui_*.py`, domain logic in `procesos/*.py`, data in `models/gestor_sqlite.py`. Reuse helpers from `models/facturas_common.py` and `utils/utilidades.py`.
- Avoid touching `main.py` unless explicitly required (contains hardcoded advisory header data).
- Operate on SQLite directly; update `plantillas.json` only if seeding is needed.
- Back up `plantillas/` folder before schema migrations.
